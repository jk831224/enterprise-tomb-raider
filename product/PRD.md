## 一句話定位

把 Claude Code 變成一個**有底線的企業情報調查員**——用對話完成原本需要花一整週、跨越十幾個資料源的產業與公司研究，並在資料不足時誠實標記，而不是用看似合理的猜測填補。

## 為什麼存在這個產品

### 觀察到的痛點

做以下決定的人，每一次都要重新拼湊一份「這家公司／這個產業到底是怎麼回事」的脈絡：

- **投資人** 評估早期標的：商工登記、團隊背景、競品定位、估值合理性
- **求職者** 評估 offer：公司穩定度、團隊結構、創辦人風格、產業前景
- **商業合作對象** 評估合作風險：營運規模、財務體質、訴訟史、董事變更頻率
- **競品分析師** 拆解對手：產品線、定價、市佔、供應鏈位置
- **盡職調查方** 為決策建立底線：是否有未揭露的關聯法人、爭議或合規問題

這些任務的共同形狀：**搜尋成本極高，但結論結構高度相似**。一個熟練的人可以在 4-8 小時做完一份，但每個人都從零開始，每次都重新踩同樣的坑（同名公司混淆、過時登記資料、年報沒讀全、漏掉非主流來源裡的關鍵線索）。

### 為什麼現有工具不夠

| 工具類型 | 不足 |
|---------|------|
| ChatGPT browsing / Perplexity | 沒有結構化研究流程；容易在內容農場迷失；無法強制交叉驗證 |
| PitchBook / Crunchbase / Bloomberg | 付費昂貴；台灣中小企業覆蓋率低；資料死板，無法回答開放問題 |
| 商工登記 / MOPS / 法院判決書 | 資料分散；不會自己拼成故事；非結構化欄位需要人工解讀 |
| 顧問報告 | 貴；交期長；不適合快速判斷 |
| 委託情報公司 | 最貴；最慢；門檻最高 |

**留下的空缺**：一個能在 30 分鐘內，**用免費的公開資料源**，**對自己誠實**地產出**有結構、可追溯**的情報報告的工具。

## 使用者與情境

### Persona

| Persona | 主要關切 | 報告會被用來決定什麼 |
|---------|---------|-----------------------|
| **投資人**（早期/天使） | 創辦人背景、團隊穩定度、市場規模、競爭護城河 | 是否進入下一輪盡調 / 直接拒絕 |
| **求職者** | 公司財務體質、團隊規模真實性、創辦人風格、爭議史 | 是否接 offer / 議價依據 |
| **商業合作夥伴** | 對方營運規模、付款能力、訴訟史、關聯法人 | 是否簽約 / 合約條件 |
| **競品分析師** | 對手產品線、定價、市佔、近期動向 | 內部 benchmarking / 戰略簡報 |
| **盡職調查方** | 全面性 + 異常標記 | 為更深入調查設定優先順序 |

> 系統目前支援的角色設定見 `.claude/skills/recon/SKILL.md` Step 0（onboarding 流程）。每次完成分析會額外產出**決策簡報（Decision Brief）**，根據使用者角色重新解讀報告重點。

### Jobs To Be Done

按使用頻率排序：

1. **「我有 30 分鐘要評估這家公司，給我足夠的資訊做下一步決定」** — 主要 JTBD
2. **「告訴我這家公司有沒有我應該擔心的事」** — 紅旗發現
3. **「這個產業現在的格局是什麼？我這家公司的位置在哪？」** — 路徑 B 的錨點分析
4. **「我已經有年報 PDF / 截圖 / 訪談筆記了，幫我把它整合成完整的觀點」** — Drop Zone JTBD（v1.4 新增）
5. **「我聽過 N 個關於這家公司的版本，哪個是真的？」** — 交叉驗證 JTBD

### 反 Persona（明確不服務的人）

- **要求即時市場資訊的人**（股價分鐘級變動、輿情監控）→ 不是這個產品的長處
- **要做網路爬蟲的人**（大量、結構化、自動化）→ 應該寫專用爬蟲
- **要做內容農場式 SEO 摘要的人** → 這個產品有意拒絕「填滿就好」的取向

## 設計原則（不可妥協）

這些原則貫穿所有 RFC 和設計決策。違反原則的功能即使技術上可行也不做。

### 1. 絕對誠實（Honesty as Default）

找不到的數據標記 `[資料缺失]`，不以看似合理的猜測填補。寧可留白也不造假。**這是系統的底線，任何階段、任何規模、任何情況都不得違反。**

> 出處：`agent/AGENT-CORE.md` 核心原則第二條

### 2. 強制交叉驗證

關鍵欄位（董監事、資本額、地址、員工數）至少兩個獨立來源比對一致才能寫入報告。單一來源不可直接採信。

> 出處：v1.2 引入後，貫穿至今。`agent/prompts/entity-verification.md`

### 3. 資料缺失明示為一級產出

「我們搜不到什麼」本身就是高價值的訊號（可能代表 governance 問題、可能代表規模太小、可能代表刻意隱藏）。`[資料缺失]` 不是失敗，是結論的一部分。

### 4. Drop Zone 優先（v1.4 新增）

使用者主動提供的檔案視為最高優先級資料源，但**不豁免交叉驗證**。詳見 [`RFC-001`](rfcs/RFC-001-drop-zone.md) 與 `references/methodology/drop-zone.md`。

### 5. 數據源優先序

```
使用者提供文件（cases/{target}/input/）
  > 年報 / 財報
  > 官方 IR 網站
  > 權威財經平台
  > 新聞報導
  > 搜尋 snippet
```

### 6. 規模自適應

微型 / 中型 / 大型上市公司用不同的分析框架。微型企業硬套三大報表只會產出猜測，改用現金流結構分析；大型上市公司年報是 must-have。

> 出處：`references/methodology/scale-classification.md`

### 7. 對使用者透明的成本

每次分析開始前都做**預分析評估**：告訴使用者預估搜尋次數、預估時間、推薦模型。不讓使用者在不知情的情況下消耗配額。

> 出處：v1.3 引入。`.claude/skills/recon/SKILL.md` Step 4.0.5

## 目標 vs 非目標

### Goals

| 目標 | 衡量方式 |
|------|---------|
| **G1. 結構化、可追溯** | 每份報告都符合 `output-quality.md`，每個論點都有來源 |
| **G2. 證據等級可信賴** | 四級標記分布合理；驗收時不出現「應為缺失但寫成事實」的錯誤 |
| **G3. 交叉驗證有效阻擋污染** | 自 v1.2 引入後零起「過時登記資料寫入報告」事件 |
| **G4. 規模自適應產出深度合理** | 大型公司報告深度 ≥ 中型 ≥ 微型，但微型不被「框架硬套」浪費 token |
| **G5. 對使用者透明** | 預分析評估在 1 輪內呈現完整成本結構 |
| **G6. 迭代過程本身可審視** | 所有設計變更都有 RFC，所有效能影響都有量測或估算 |

### Non-goals

| 拒絕做 | 為什麼 |
|--------|--------|
| 即時市場資訊 / 股價 | 不是這個產品的長處，會引入幻覺風險 |
| 自動化批次抓取多家公司 | 會破壞「對話 + 規模自適應」的流程設計，且容易踩 robots.txt |
| 突破付費牆 / 反爬蟲 | 違反原則 1（誠實）和 fetch-policy 的黑名單設計 |
| 訓練專用模型 | 不必要；Claude + 結構化 prompt + 知識層已足夠 |
| 美觀的 UI | CLI 是 feature 不是 bug，目標使用者熟悉終端 |
| 多語言介面 | 暫時聚焦中文 + 英文資料源 |

## 成功指標

### 系統層級（每次大版本更新時量測）

| 指標 | 定義 | 目標 | 量測方式 |
|------|------|------|---------|
| **報告完整性** | 模板欄位中非 `[資料缺失]` 的比例 | 大型 ≥85%、中型 ≥70%、微型 ≥50% | 對 baseline 案例重跑後人工檢查 |
| **證據充分度** | 充分證據 / (充分+部分) 的比例 | ≥60% | 同上 |
| **交叉驗證命中率** | 關鍵欄位有 ≥2 獨立來源比對的比例 | 100%（強制） | 自動規則檢查 |
| **平均 token 成本** | 一份完整報告的 token 用量 | 中型 ≤ baseline-v1.1 ×1.2 | 從 perf log 取 |
| **平均完成時間** | 從輸入到報告產出 | 微型 15-20 分；中型 20-25 分；大型 25-35 分 | 計時 |
| **搜尋命中率** | 「被引用的搜尋結果 / 總搜尋呼叫」| 待 RFC-001 實測後設定 baseline | 從 trace 估算 |

### 使用者體驗層級（質性，靠 case 沉澱觀察）

- 預分析評估後**使用者直接說「繼續」**的比例（高表示成本透明度足夠）
- 報告產出後使用者**追問「為什麼這個是缺失」**的比例（高代表降級機制不夠透明）
- Drop Zone 使用率（v1.4 後追蹤）

### 候選下一步（無排序，待 RFC 評估）

| 候選功能 | 一句話描述 | 估計影響 |
|---------|----------|---------|
| **Adversarial cross-checking** | 強制 agent 為每個結論找一個反證或質疑來源 | 提升 G2 證據可信度，可能 +30% 搜尋成本 |
| **多公司比較模式** | 一次比較 2-3 家同產業公司，產出對照表（v1.6 Registry 提供資料基礎） | 新 JTBD，獨立於現有路徑 |
| **訪談腳本生成** | 報告產出後額外產出「該追問什麼」的問題清單 | 補強 Decision Brief |
| **資料缺失章節獨立** | 把散落的 `[資料缺失]` 標記彙整成 first-class output | 直接呼應原則 3 |
| **Inline cost dashboard** | 執行中即時顯示已消耗的搜尋/fetch 預算 | 補強原則 7 |
| **韓國 DART / 日本 EDINET 整合** | 擴展年報解析到日韓上市公司 | 擴大 G4 覆蓋規模 |
| **Drop Zone OCR** | 支援 HEIC、影像類年報的文字提取 | 補強 v1.4 體驗 |

### 開放問題

- 如何量化「Decision Brief 對使用者實際決策的影響」？目前沒有 instrumentation
- `[資料缺失]` 的「合理留白率」應該是多少？太少代表幻覺，太多代表降級過早
- Drop Zone 是否該支援多目標（同一份產業報告餵入多家公司的年報）？
