# Performance Tracking 方法論

> 這個資料夾追蹤每次設計變更對系統效能的影響。報告必須**輕量、可比較、誠實**——不寫 200 行的散文，寫 30-50 行的對照表。

## 為什麼追蹤效能

每次新功能都可能換來成本：更多 token 預載、更多 file read、更多 web 搜尋、更長完成時間。如果不追蹤，「持續迭代優化」會變成「持續迭代但越來越慢」。

效能追蹤的目的不是**最佳化每一個 byte**，而是：

1. 在做設計決策時有量化的代價估算
2. 在實作後驗證估算是否正確
3. 在累積足夠資料後找出**真正的瓶頸**（通常不是你以為的那個）
4. 作品集呈現時有可信的數字背書「我確實知道我的系統在做什麼」

## 追蹤的維度

### 一級指標（每個 RFC 必填預估，重要變更必量測）

| 維度 | 定義 | 為什麼重要 |
|------|------|----------|
| **預載 token** | system prompt + skill + agent core + 階段 prompt 的合計 token 數 | 直接影響每次對話的 base cost；context window 預算 |
| **平均搜尋次數** | 一份完整報告所需的 web_search 呼叫數 | 主要的對外成本來源 |
| **平均 fetch 次數** | web_fetch 呼叫數 | 較貴的呼叫，且容易撞 Cloudflare |
| **平均 file read 次數** | Read 呼叫數 | 內部成本，但累積會壓 context |
| **完成時間** | 從輸入到報告產出的牆鐘時間 | 使用者體驗 |

### 二級指標（質性，挑值得的 RFC 才量）

| 維度 | 定義 | 量測方式 |
|------|------|---------|
| **搜尋命中率** | 被報告引用 / 直接導致下一步驗證的搜尋呼叫數 ÷ 總搜尋呼叫數 | 從 trace 估算，0-1 區間 |
| **報告完整性** | 模板欄位中非 `[資料缺失]` 的比例 | 人工檢查 |
| **證據等級分布** | 充分證據 / 部分證據 / 推測 / 缺失 各占比 | 自動規則檢查 |
| **降級觸發次數** | 階段中觸發降級策略的次數 | 從 trace 統計 |

## 命中率（Hit Rate）的操作型定義

「命中率」這個詞容易模糊，這裡明確界定：

### 分子：被視為「命中」的搜尋呼叫

一次 `web_search` 或 `web_fetch` 呼叫被視為命中，當它**至少符合一項**：

1. **直接引用**：搜尋結果中至少一個來源被寫進最終報告（出現在引用列表）
2. **驗證鏈接**：搜尋結果觸發了下一次 fetch / search，最終形成被引用的證據鏈
3. **負面確認**：搜尋結果讓 agent 確認某個假設為偽，並寫入 `[資料缺失]` 或衝突清單

### 分母：所有搜尋呼叫

包括完全沒結果的、搜出無關內容的、被 fetch 失敗的。

### 不算命中

- 同一資訊重複搜尋的後續呼叫（去重）
- 純探索性搜尋但沒導向任何結論
- 被 fetch policy 黑名單擋住的呼叫

### 命中率的解讀

| 命中率 | 解讀 |
|--------|------|
| < 0.5 | 搜尋策略太發散，需要更窄的 query 或更多 drop zone 餵資 |
| 0.5–0.7 | 健康範圍 |
| > 0.7 | 搜尋很精準，但可能代表「保守不冒險」，要檢查是否因此漏掉非主流來源 |
| > 0.9 | 警訊：可能在 confirmation bias 下只搜已知答案 |

## 量測方式

### 設計時預估（每個 RFC 必填）

填在 RFC 的 §6「設計時預估」表格。不求精準，求**可比較**。估算依據：

- **預載 token**：用檔案 byte 數 × 中英混合 token 比率（取 1.3）
- **新增 Read/Glob**：直接從 SKILL.md / prompt 的程式邏輯數
- **預期搜尋次數變化**：基於現有 baseline ± 主觀判斷
- 預估數字寫進 RFC 後**不再改**，讓未來實測能校正主觀判斷

### 實測（重要 RFC 才做，純文件變更不需要）

**何時做實測**：
- 變更觸及搜尋迴圈、預算、資料來源優先序
- 變更引入新的階段或大幅改寫既有階段
- CHANGELOG 在版本 release 前

**何時不做**：
- 純文件、註解、命名變更
- 只影響極端情況的修補
- 與其他已實測 RFC 同一批 commit、效應重疊

**怎麼做**：
1. 跑一個（或最多兩個）真實案例，最好對照 baseline 同一家公司
2. 從 session 結束後的對話紀錄回溯：搜尋呼叫數、fetch 呼叫數、Read 數、產出 byte 數
3. token 數用估算（Claude 不會自報用量）：1 中文字 ≈ 1.3 token
4. 把數字填進 RFC 的 §6「實測結果」表格
5. 偏差 > ±30% 的維度要在 RFC 裡解釋為什麼

## 常見陷阱

- **不要把效能報告寫成自誇散文**。數字優先，敘述次之。
- **不要只測「最佳案例」**。挑代表性案例，包括規模分類各一個。
- **不要在沒對照的情況下宣稱「優化」**。沒有 baseline 就不是優化，是改寫。
- **不要追求精準**。token 估算 ±20% 是 OK 的，重點是趨勢一致。
- **不要每個 RFC 都實測**。挑值得的，避免測試成本超過設計成本。
