## 1. Summary

`product/perf/README.md` 對命中率（Hit Rate）的單維度定義會產生**假陽性警訊**：高聚焦調查的案例命中率自然會 > 0.9，但這不代表 confirmation bias。本 RFC 提議引入第二個維度「探索性搜尋比例」（Exploration Ratio），讓「高命中率 + 高探索性」可以與「高命中率 + 零探索性」明確區分，避免實測報告中浪費篇幅解釋「為什麼這次警訊不是真警訊」。

## 2. Background / Problem

### 觸發案例

RFC-001 首次實測（雲端資安代理商 B 2026-04-08）的命中率估算為 **0.93**。按 `product/perf/README.md` 現行區間表：

| 命中率 | 解讀 |
|--------|------|
| < 0.5 | 搜尋策略太發散 |
| 0.5–0.7 | 健康範圍 |
| > 0.7 | 搜尋很精準，但可能代表「保守不冒險」 |
| **> 0.9** | **警訊：可能在 confirmation bias 下只搜已知答案** |

0.93 直接命中「警訊」區間。但實測報告分析後發現，這次的高命中率**真實成因**是：

1. Entity verification 階段的高聚焦搜尋（每個 query 都目標明確）
2. 計畫外的高價值發現（關聯法人「系統整合商 B-Affiliate」）拉高了「實際被引用」的比例
3. 員工數搜尋的多次「無結果」也算「負面確認」（按 perf README 的定義），拉高了命中率分子
4. 確實只有 2 次純探索性、無導向結論的呼叫

**真正的 confirmation bias 應該長這樣**：
- 大量重複搜尋已知答案的不同 phrasing
- 從不搜尋反證
- 從不搜尋與假設相反的關鍵詞

而本次實測**不符合**任何上述模式 — 是高聚焦但非偏見。

### 現況的限制

單維度命中率有兩個結構性問題：

1. **假陽性**：高聚焦調查 ≠ confirmation bias，但單維度無法區分
2. **沒有針對「探索」的鼓勵**：如果未來有人為了避免警訊而**故意降低命中率**（多做無意義搜尋），反而會讓品質下降

實測報告 §「對 perf README 命中率區間的回饋」也已建議：「在命中率定義中加入第二維度：探索性搜尋的比例」。本 RFC 將這個建議正式化。

## 3. Goals / Non-goals

### Goals

- **G1**：消除「高命中率 + 高聚焦調查」的假陽性警訊
- **G2**：明確區分 confirmation bias（壞）vs 高聚焦調查（好）
- **G3**：對「主動探索反證 / 反向假設」這個行為提供量化獎勵
- **G4**：保持輕量 — 量測成本不超過現行單維度的 1.5 倍
- **G5**：可向後相容 — 既有 baseline 報告（v1.1 個人理財 SaaS A案）不需要重新計算

### Non-goals

- 不重寫整個 perf 量測框架
- 不引入需要 LLM 自評的新指標（增加成本）
- 不改變 RFC-001 的實測結果（已記錄為歷史值，留作對照）
- 不要求每個 RFC 都用新指標（仍是「挑值得的 RFC 才量測」原則）

## 4. Options Considered

### Option A：保留單維度，提高警訊閾值

把警訊區間從 `> 0.9` 改為 `> 0.95`，並把 `0.9–0.95` 標為「視情況解讀」。

**優點**：
- 改動最小（一行表格）
- 完全向後相容
- 零實作成本

**缺點**：
- 沒解決根本問題：依然是單維度
- 「視情況解讀」是 escape hatch，未來實測報告仍要每次解釋
- RFC-001 實測的 0.93 仍會卡在模糊區
- 對「鼓勵探索」這個目標沒有任何貢獻

### Option B：引入第二維度「探索性搜尋比例」（Exploration Ratio）

定義第二指標 **Exploration Ratio**：
```
Exploration Ratio =
  (主動搜尋反證 + 主動搜尋假設外的關鍵詞 + 主動展開非預期線索的搜尋次數)
  ÷ 總搜尋呼叫次數
```

組合判讀矩陣：

| 命中率 | 探索比例 | 解讀 |
|--------|---------|------|
| 高（>0.85）| 高（>0.2）| ✅ **健康聚焦調查** — 高效命中且持續探索 |
| 高（>0.85）| 低（<0.1）| ⚠️ **真正的 confirmation bias 警訊** |
| 中（0.5-0.85）| 高 | ✅ 標準健康範圍 |
| 中（0.5-0.85）| 低 | ✅ 標準健康範圍（聚焦執行模式）|
| 低（<0.5）| 高 | ⚠️ 探索過度、搜尋策略太發散 |
| 低（<0.5）| 低 | 🔴 系統性問題 — 既不命中也不探索 |

**優點**：
- 直接解決假陽性問題
- 對 RFC-001 實測的「0.93 + 主動追關聯法人」可以正確標為「健康聚焦調查」
- 引入「探索」作為一級指標，符合系統「資料缺失明示為一級產出」的精神
- 對 baseline 向後相容：可以選擇性地對歷史 baseline 補算 Exploration Ratio，但不強制

**缺點**：
- 量測成本增加：需要對每次搜尋呼叫做「是否為探索」的標記
- 「探索性」的定義有主觀性，標記時可能不穩定
- 報告格式變動，現有 perf README 需更新

### Option C：用 LLM-as-judge 自評整體調查品質

讓 Claude 自己讀完整個調查 trace，回答「這次調查是否有 confirmation bias 跡象」（0-1 分數）。

**優點**：
- 不需要對每個呼叫單獨標記
- 能捕捉到更細緻的 nuance（例如「沒有搜尋反證但所有結論都對」）

**缺點**：
- **顯著增加成本**：每個 perf 報告多一次 LLM 呼叫
- 違反 G4「量測成本不超過 1.5 倍」
- LLM 自評有 alignment 問題（傾向給自己高分）
- 違反 perf README「不要把效能報告寫成自誇散文」原則

### Option D：用「相對排名」取代絕對閾值

不設絕對閾值，而是讓每個案例的命中率與**該案例規模分類的歷史平均**比較（例如「微型企業案例的命中率歷史平均 0.75，本次 0.93 高出 24%」）。

**優點**：
- 自動適配不同案例複雜度
- 沒有「假警訊」問題

**缺點**：
- 需要先累積足夠 baseline（目前只有 1 個 baseline + 1 個實測，樣本太小）
- 「歷史平均」會隨時間漂移，不適合做長期決策
- 對單一案例的解讀沒有絕對標準

## 5. Decision

**選擇**：Option B（引入探索性搜尋比例第二維度）

**理由**：

- 直接解決根本問題（假陽性），而不是 escape hatch（A）
- 量測成本可控（B 的 +0.5x 開銷 < C 的 +1x LLM 呼叫）
- 符合系統設計哲學（探索與誠實同樣重要）
- 對未來的 perf 報告提供結構化判讀工具，減少每次手動解釋

### 關鍵決策點

| 決策 | 選擇 | 原因 |
|------|------|------|
| 探索比例的計算粒度 | 每個搜尋呼叫做標記（探索 / 非探索）| 簡單清楚，不需要主觀打分 |
| 標記時機 | 寫 perf 報告時回溯標記，非搜尋當下 | 避免影響執行；perf 報告本身就是回溯 analysis |
| 探索的定義 | 「主動搜尋反證 / 反向假設 / 計畫外線索」三類 | 涵蓋真正想鼓勵的行為，排除「無意義廣搜」|
| 是否強制每個 perf 報告都計算 | **否**，仍是「挑值得的才量」原則 | 與現行 perf methodology 一致，避免過度量化 |
| 是否回填歷史 baseline | 否，新的 perf 報告開始才用 | 避免考古成本，baseline 依然作為相對對照 |

## 7. 風險

| 風險 | 嚴重度 | 緩解方式 |
|------|--------|---------|
| 「探索」標記主觀性導致跨報告不可比 | 中 | 在 perf README 加入 3-5 個明確範例（什麼算探索、什麼不算）|
| 為了拉高探索比例而做無意義搜尋 | 中 | 強調「探索 = 主動搜尋反證或非預期方向」，純廣搜不算 |
| 增加 perf 報告寫作摩擦，降低願意寫的頻率 | 中高 | 保持「重要 RFC 才量」原則；提供標記模板降低成本 |
| 雙維度判讀矩陣比單維度複雜，新讀者難理解 | 低 | 在 perf README 用視覺化矩陣呈現 |
| 既有 baseline 不可比 | 低 | 已在 G5 明確不要求回填 |

- 任何 agent prompt — 命中率是事後評估指標，不影響 agent 執行邏輯

### 探索性搜尋的判定範例（將寫入 perf README）

| 範例 | 算探索嗎？|
|------|---------|
| 搜尋「雲端資安代理商 B 訴訟」明確尋找負面紀錄 | ✅ 算（主動搜尋反證）|
| 在發現關聯法人後搜尋「系統整合商 B-Affiliate 雲端資安代理商 B」延伸 | ✅ 算（追計畫外線索）|
| 搜尋「雲端資安代理商 B 客戶 案例」期待特定案例 | ❌ 不算（confirming 已知方向）|
| 搜尋「[人名] 判決書」做標準合規檢查 | ✅ 算（系統性反證搜尋）|
| 重複用不同 phrasing 搜「雲端資安代理商 B 員工數」3 次 | ❌ 不算（同方向重複）|

## 9. Followups / Open Questions

- **F1**：Exploration Ratio 的 0.2 / 0.1 閾值是否需要調整？需累積 3-5 個 perf 報告後檢視
- **F2**：是否需要把 Exploration Ratio 加入主報告的「附錄 A 搜尋紀錄摘要」（讓使用者也能看），還是只放在 perf 報告？
- **F3**：是否值得做一個半自動的「標記輔助工具」（例如從 perf 報告中提取 search query 列表，agent 協助標記）？
- **F4**：當有 5+ 個 perf 報告後，可考慮把雙維度的判讀矩陣替換成「命中率 vs 探索比例的散點圖」（每個案例一個點，視覺化判讀）
- **F5**：如果 RFC-001 §9 F1（補測 drop zone 帶內容情境）執行，可以同步用 Option B 新定義驗證雙維度的可行性
